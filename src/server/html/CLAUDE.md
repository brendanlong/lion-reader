# HTML Sanitization (`src/server/html/`)

This file governs the entry-HTML sanitization pipeline: the native sanitizer (`native/sanitizer/`, exposed as `@lion-reader/sanitizer`), its thin TypeScript wrapper `sanitize.ts`, the read-path family sanitizer `sanitize-entry.ts`, and their callers. **Read this before touching anything here — this code is security-critical.**

## Why it's security-critical

Entry bodies (and AI summaries) are rendered via `dangerouslySetInnerHTML`, so untrusted HTML is sanitized **on the server** with `sanitizeEntryHtml` from `@/server/html/sanitize`. The sanitizer is the primary XSS defense on a surface that renders untrusted HTML, so treat every change to the native crate's allow-lists/transforms as security-sensitive. (The `session` cookie is `httpOnly` — see "Session Cookie" in `src/server/auth/CLAUDE.md` — so an XSS here is no longer a one-step session-token theft, but XSS remains a serious vulnerability regardless.) The client renders trusted HTML and ships no sanitizer. Sanitization runs **on every read**: entries store only the raw content columns and the read chokepoints sanitize them before returning. Those chokepoints live in the **services layer**: `toFullEntry`/`getEntry`/`getEntries` in `src/server/services/entries.ts` (covering tRPC `entries.get` + `entries.fetchFullContent`, MCP `get_entry`, Google Reader, and Wallabag), the saved-article read paths in `saved.ts`, `fetchAndStoreFullContent` in `full-content.ts`, and summary generation in `summarization.ts` — all funnel through `sanitizeEntryContentFamily` (`sanitize-entry.ts`) or `sanitizeEntryHtml`/`sanitizeEntryHtmlAsync`. Do **not** reintroduce a client-side sanitizer, and do **not** persist sanitized output back to the DB. Feed `summary`/`title`/`author` are rendered as escaped text, not HTML; keep them that way.

## The native sanitizer (`native/sanitizer/`)

The whole pipeline is a Rust N-API module (built by `pnpm build:native`, a plain `cargo build` — no `@napi-rs/cli`): one call into native code runs, in order,

1. **MathJax CHTML → MathML** (`core/src/mathjax.rs`) — sources like LessWrong deliver math pre-rendered as `<mjx-*>` elements whose glyphs live in CSS class names; the sanitizer would strip them, so each located `<mjx-container>` substring is parsed (html5ever) and rewritten to presentation MathML (allow-listed, renders natively). Assistive MathML (`<mjx-assistive-mml>`) is preferred verbatim when present. Unknown `mjx-*` wrappers are unwrapped and reported via `warnings` (logged by the TS wrapper — the canary for MathJax layout drift). Degrades to "math stripped" on error.
2. **Inline SVG** (`core/src/svg.rs`, issue #923) — each top-level `<svg>` is located by byte range, parsed with html5ever's spec foreign-content handling (camelCase tag/attribute adjustment exactly as a browser does), sanitized against the DOMPurify-derived allow-list (no `script`/`foreignObject`/`use`/animation/`style`; `href` scheme-validated per element), and swapped for an opaque per-call-nonced placeholder that the main pass passes through as inert text, then reinserted afterwards. Degrades to "SVG stripped" on error.
3. **The HTML allow-list pass** (`core/src/sanitize.rs`) — the core allow-list on **lol_html** (streaming, no tree). Unknown tags are unwrapped (children kept); the `DROP_WITH_CONTENT` set is removed whole (subtree and all); attributes are allow-listed (globals + `data-*`/`aria-*` + MathML + per-tag); URL attributes are scheme-checked on the decoded value; external links get `target="_blank" rel="noopener noreferrer"`, images `loading="lazy"`, iframes survive only as normalized embeds (below). The load-bearing rule when editing this: **any new rawtext/RCDATA element (`title`/`xmp`/`noscript`/…) MUST go in `DROP_WITH_CONTENT`, never the unwrap path** — inside rawtext the tokenizer reads contents as text, so unwrapping re-emits markup verbatim for the browser to re-parse as live (mXSS). The `DROP_WITH_CONTENT` / `GLOBAL_ATTRS` / `attr_allowed` definitions carry the full lists and per-entry rationale (why MathML `annotation`/`annotation-xml` drop, why `role`/`aria-*` are kept) — read them there, not here.
4. **SVG re-insertion.**

Element/attribute byte preservation: lol_html does **not** re-serialize untouched markup — kept tags/attributes/text pass through byte-identical (uppercase tags stay uppercase, entities stay encoded). It tokenizes per the HTML5 spec — the same tokenization a browser applies to the output — which eliminates the parser-differential (mXSS) attack class.

**Iframe embeds** (`core/src/embeds.rs`, issue #922): block-by-default, opt-in per provider (YouTube → youtube-nocookie, Vimeo, Spotify, SoundCloud, Bandcamp, CodePen). The src is validated against the provider's hosts + a strict path regex and rebuilt from scratch on the canonical host with an allow-list of query params; the sanitizer forces per-provider `sandbox`/`allow`. `normalizeEmbed` is exported to JS for tests and for TS callers that synthesize embeds (the YouTube plugin builds iframes the sanitizer must keep accepting — there's a sync-guard unit test).

**Scanner** (`core/src/scanner.rs`): the byte-range locator used by the MathJax/SVG passes (comment/rawtext/quote-aware, depth-tracked; unclosed `mjx-container`s recover absorbed article content at the last explicit `</mjx-math>` close, unclosed `<svg>`s extend to EOF). Its guard: a located range must actually parse to the target element or it's spliced through verbatim, and everything flows through the sanitize pass afterwards, so a mislocation can mangle content but never emit unsanitized HTML.

Rust unit tests live in the crate (`cargo test -p lion-reader-sanitizer-core` from `native/sanitizer/`); TS-side pipeline tests in `tests/unit/sanitize-entry-html.test.ts`, `mathjax-chtml.test.ts` (real mathjax-full 3.2.2 fixtures), `sanitize-svg.test.ts`, `embed-providers.test.ts`.

## Per-read sanitization (no persisted columns)

Sanitization is **not persisted** (issue #1282). Entries store only the raw content columns (`entries.content_*` / `full_content_*`); every read sanitizes them on the fly. The native sanitizer is fast enough to make this cheap — ~0.09 ms / 10 KB, ~6 ms for a 700 KB math-heavy body, and off the event loop for large bodies (see below).

The single read-path chokepoint is `sanitizeEntryContentFamily(family, { original, cleaned })` in `sanitize-entry.ts`. A content "family" is an `(original, cleaned)` pair:

- The **content** family sanitizes **both** variants — the frontend has a user-facing original/cleaned toggle for feed content (`hasBothVersions` in `EntryContentBody.tsx`).
- The **full-content** family's serving rule is strictly `cleaned ?? original` (no toggle), so the (whole raw page) `original` is sanitized **only when `cleaned` is NULL** — otherwise it is never displayed, so we skip the whole-page sanitize and return `null`. Consumers already fall back `cleaned ?? original`.

`SANITIZER_VERSION` (defined in the native crate `core/src/lib.rs`, re-exported by `sanitize.ts`) is **purely informational**. Because nothing stores a sanitized copy, a change to the allow-lists/transforms takes effect on the next read everywhere — **a rules change is just a deploy**, no version bump required.

## Event-loop protection (async sanitize)

The native module exposes both sync and async forms. `sanitizeEntryHtml` is synchronous (~1ms/100KB); `sanitizeEntryHtmlAsync` runs the same pipeline on the **libuv thread pool** for bodies above ~10 KB (smaller ones run inline — cheaper than scheduling a task), so app-server request paths never block the event loop on a large body. `sanitizeEntryContentFamily` calls `sanitizeEntryHtmlAsync`, and the read paths use it: `getEntry`/`toFullEntry` sanitize per entry, and `getEntries` (the Google Reader stream-contents batch, up to ~100 entries) sanitizes with a small concurrency bound (`GET_ENTRIES_SANITIZE_CONCURRENCY` in `entries.ts`) so a large-body batch can't flood the pool. **Background jobs deliberately use the synchronous `sanitizeEntryHtml`** where they sanitize (note the feed-fetch/email-ingest **write** paths don't sanitize at all — they store raw). Readability-style article extraction is also native (`@lion-reader/readability`, the dom_smoothie crate — a Rust port of Mozilla Readability) and follows the same pattern: `cleanContent` (in `@/server/feed/content-cleaner`) is synchronous for background jobs, and request paths use `cleanContentAsync`, which runs the extractor on the libuv thread pool with the same ≤10 KB inline shortcut (`fetchFullContent` takes an `offloadClean` flag, defaulting on but set false by the feed worker).

## AI summaries

AI summaries are the one place sanitized HTML is still stored (`entry_summaries.summary_text`): they're generated once, by us, so they're written already-sanitized at generation time (`summarization.ts`). Even so, the summary read path re-sanitizes with the current rules on each read (`sanitizeEntryHtml(record.summaryText)` in the summarization router) — summaries are small, so re-sanitizing per read is cheaper than tracking staleness, and it means an allow-list change applies to stored summaries too without a migration.
